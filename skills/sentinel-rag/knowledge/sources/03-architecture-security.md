For cybersecurity, this means representing CVE → CWE → CPE relationships as graph edges, enabling:\n- **Relationship discovery**: Predict unknown CVE-to-CWE associations (Mean Reciprocal Rank 0.85+)\n- **Multi-hop reasoning**: \"This CVE affects Product X via CWE-123, which also affects Product Y\"\n- **Community detection**: Group related vulnerabilities by attack vector/industry\n\n**Research Backing:**\n- Academic paper \"Uncovering CWE-CVE-CPE Relations with Threat Knowledge Graphs\" (ACM TOPS, 2024-2025) demonstrates that knowledge graphs can predict unreported vulnerabilities\n- CTINEXUS (2026) shows LLM-optimized systems achieving few-shot relation prediction in CTI reports\n- GraphRAG implementations with Neo4j achieve 92.6% composite accuracy against baseline RAG methods\n\n**Tools Ecosystem:**\n- **LlamaIndex**: GraphRAGStore class wraps Neo4j Property Graph, includes community-based summarization\n- **Neo4j**: Native graph database with Cypher query language, supports version control via snapshots\n- **NetworkX**: Python library for graph analysis (local use, not production-scale)\n\n**For Sentinel (6000 docs):**\n- **Cost/Benefit**: Worth implementing for ~115 cybersecurity rules + ~2400 CVE documents\n- **Implementation effort**: 2-3 weeks (define entity schema, build extraction pipeline, index into Neo4j)\n- **Incremental adoption**: Start with vector search (current ChromaDB), add graph layer later\n- **ROI**: Enables queries like \"Show me all zero-day mitigations that also prevent supply-chain attacks\" without fine-tuning\n\n**Anti-pattern:** Building a graph without clear entity/relationship taxonomy = semantic soup. Define your schema (CVE, CWE, CPE, Attack Pattern, Mitigation, Industry) before indexing.\n\n---\n\n### 1.2 Agentic RAG: Dynamic Reasoning\n\n**State of the Art (2025-2026):**\n\nAgentic RAG embeds an autonomous agent that decides _when_ to retrieve, _what_ to retrieve, and _how_ to validate results. Four key patterns:\n\n1. **Reflection**: Agent pauses to critique its reasoning (\"Is my answer grounded in retrieved docs?\")\n2. **Planning**: Agent breaks complex queries into subqueries with execution order\n3. **Tool use**: Agent invokes retrieval, web search, code execution based on task\n4. **Multi-agent collaboration**: Specialized agents (CVE lookup, CWE mapper, remediation finder) coordinate\n\n**2025-2026 Benchmarks:**\n- **RAG-EVO** (EPIA 2025): Evolutionary learning achieves 92.6% composite accuracy\n- **A-RAG** (arXiv:2602.03442, Feb 2026): Keyword + semantic + chunk-level tools improve QA by 5-13%\n- **Reasoning RAG**: System 2 reasoning (slow, thorough) outperforms System 1 (fast, pattern-matching) on adversarial queries\n\n**Implementation Frameworks:**\n- **LangChain + LangGraph**: Modular components + state persistence + loops\n- **Multi-agent patterns**: Independent agents query different KB sections in parallel, merge results\n\n**For Sentinel:**\n- **Quick win**: Implement query classification agent (\"Is this CVE ID lookup?\" vs \"Tell me about auth vulns\")\n- **Medium-term**: Add self-correction loop (agent validates retrieved docs against rules schema)\n- **Advanced**: Multi-agent orchestration (web-agent queries GitHub advisories, cve-agent queries NVD, rule-agent validates against KB)\n\n**Trade-off**: Agentic systems are slower (multiple LLM calls) but more accurate for complex security queries. Worth it for user-facing `/security` skill.\n\n---\n\n### 1.3 Self-RAG & Corrective RAG\n\n**Self-RAG (Train model to self-critique):**\n- Model generates special tokens: `[Retrieve=Yes]` (do I need more context?) and `[ISREL=Irrelevant]` (is this doc useful?)\n- Eliminates external evaluator, reduces latency\n- Best for models you can fine-tune (not applicable to Claude via API)\n\n**Corrective RAG (5-agent pipeline):**\n1. **Context Retrieval**: Fetch initial docs from vector DB\n2. **Relevance Evaluation**: Score retrieved docs (use keyword overlap + semantic similarity)\n3. **Query Refinement**: Rewrite query if relevance score < threshold\n4. **External Knowledge**: Fall back to web search or alternative sources\n5. **Response Synthesis**: Aggregate validated information\n\n**For Sentinel:**\n- Corrective RAG is immediately implementable: add relevance thresholds to your query pipeline\n- Current architecture: Query → Grep patterns + Read rules.json. Add: Relevance check before synthesis\n- Cost: 1 extra LLM call per query (to evaluate relevance). Measurably improves accuracy for out-of-KB queries\n\n---\n\n### 1.4 RAG + Fine-tuning: The Hybrid Approach\n\n**State of the Art:**\n\nThe industry consensus (2025) is **RAG-only for dynamic data**, **fine-tuning-only for style**, **hybrid for reasoning-heavy domains**:\n\n| Approach | Use Case | Cost | Latency | Accuracy |\n|----------|----------|------|---------|----------|\n| RAG-only | \"Latest CVE exploits\" | Low | ~100ms | 70-80% (hallucinations) |\n| Fine-tuning-only | \"Domain-specific terminology\" | High | None | 85-90% (stale after 3mo) |\n| **Hybrid (RAFT)** | \"Cybersecurity audits\" | Medium | ~150ms | 90-95% (reasoning + fresh data) |\n\n**RAFT (Retrieval-Augmented Fine-Tuning):**\n- Fine-tune model on security domain data (OWASP, CWE descriptions, remediation patterns)\n- Deploy fine-tuned model in RAG architecture\n- Model learns _how to reason_ about security, RAG provides _current context_\n\n**For Sentinel:**\n- **Now**: Pure RAG with ChromaDB (no fine-tuning)\n- **Next quarter**: Collect top-10 user queries + LLM-generated security audits. Use to fine-tune (optional, if Claude API allows)\n- **Expected gain**: 5-10% accuracy improvement on security classification tasks\n\n---\n\n## 2. QUERY EXPANSION & OPTIMIZATION\n\n### 2.1 HyDE: Hypothetical Document Embeddings\n\n**What it does:**\n1. Take user query: \"How do I prevent SQL injection?\"\n2. LLM generates hypothetical document (fake answer that might address query)\n3. Embed hypothetical doc into vector space\n4. Use embedding for retrieval (doc-to-doc similarity, not query-to-doc)\n\n**2025-2026 Performance:**\n- Improves precision by **40+ percentage points** on certain datasets\n- Works especially well for question-answering (not exact lookup)\n- Zero-shot: No training required\n\n**Cost Analysis:**\n- 1 extra LLM call per query (generate hypothesis)\n- ~5-10ms latency addition\n- Vector db query unchanged\n- ROI: Recommended for semantic queries (\"What are CWE mitigations?\"), not for exact ID lookups (\"Find CVE-2026-1234\")\n\n**For Sentinel:**\n- **Implementation**: Add to query preprocessing pipeline\n- **Configuration**: Disable for queries containing CVE/CWE IDs (exact match), enable for semantic queries\n- **Expected improvement**: +15-20% recall on mitigation searches\n\n---\n\n### 2.2 Multi-Query Retrieval\n\n**Approach:**\n1. User query: \"SQL injection prevention\"\n2. LLM generates 3-5 variants:\n   - \"SQL injection mitigation techniques\"\n   - \"Parameterized queries to prevent SQL injection\"\n   - \"OWASP A03:2021 SQL Injection\"\n3. Search all variants, merge results (dedup by relevance score)\n\n**Performance (2025):**\n- Improves coverage by 10-20% on complex queries\n- Works well with hybrid search (keyword + semantic)\n\n**For Sentinel:**\n- **Quick win**: Implement for broad queries (\"authentication vulnerabilities\")\n- **Skip for**: Specific CVE lookups or CWEID searches (query variants would be repetitive)\n\n---\n\n### 2.3 Query Routing & Classification\n\n**Three-tier approach:**\n\n1. **Exact Match Detection**:\n   - If query matches `CVE-\\d{4}-\\d{4,}` → Route to CVE index (Grep + cache lookup)\n   - If query matches `CWE-\\d+` → Route to CWE index\n   - **Latency**: <10ms (regex, no LLM)\n\n2. **Domain Classification** (LLM-based):\n   - Input: Query text\n   - Output: Domain + subtype (e.g., \"WEB-OWASP\", \"API-RATE-LIMITING\")\n   - Route to specialized agent (web-audit, api-audit, etc.)\n\n3. **Semantic Routing** (Advanced):\n   - Use embedding similarity to decide: Should I query KB, GitHub advisories, or NVD?\n   - Allows heterogeneous data sources\n\n**For Sentinel:**\n- **Phase 1**: Implement exact-match routing (CVE/CWE ID detection)\n- **Phase 2**: Domain classification via LLM (route to appropriate agent in `/security` skill)\n- **Phase 3**: Semantic routing to external sources (GitHub advisories, NVD) via MCP network calls\n\n**Anti-pattern:** Routing to wrong data source. Validate routing logic against top-20 user queries before deploying.\n\n---\n\n### 2.4 Query Normalization & Autocorrection\n\n**Required for cybersecurity KB:**\n- Normalize: `CWE-79` = `CWE79` = `Cross-Site Scripting (XSS)` = `Improper Neutralization of Input During Web Page Generation`\n- Typo correction: `SQL injection` ← `SQL injeciton`, `XSS` ← `XXS`\n- Alias expansion: `CORS` ← query as `Cross-Origin Resource Sharing` + `CORS` both\n\n**Implementation:**\n- Maintain mapping table: { \"canonical_id\": [...aliases] }\n- On query: Expand aliases, search for all variants, deduplicate results\n\n**For Sentinel:**\n- **Quick win**: Build alias map from knowledge-base/domains/*.json (extract all IDs + titles)\n- **Cost**: Negligible (hashmap lookup, O(1))\n- **Benefit**: 5-10% improvement in recall for user typos\n\n---\n\n## 3.
